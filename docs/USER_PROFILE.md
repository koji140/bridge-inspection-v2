## 出力スタイルの好み

### 完成版を一度に出す

部分的な修正差分の提示を嫌う。
「ここだけ変えてください」ではなく、常に**完成版のファイル全体**を出す。

> 「私はエンジニアではないので、部分の更新をするよりも、できれば全文を一度に変更できるようなやり方だとスピードをもって進められる」

### 選択肢は補足込みで出す

選択肢を提示すると「それぞれ補足を」と求める。
最初から各選択肢のメリット・デメリット・前提条件をセットで出す。

### 仕様書・ドキュメントは詳しく書く

v1 スタイル（背景・理由・補足・注意・例）が好み。
表だけで済ませない。後で読み返したときに判断の経緯まで追えることを重視する。

> 「たんぱくになりすぎた。もっとv1をベースにしてほしい。こんなにデフォルメしなくてもいい。後で読み返した時に理解が進むようにしてほしい。」

## 判断・優先順位の傾向

### 操作ステップが増えることへの強い拒絶

UX の手間が増える提案には「めちゃくちゃめんどくさい」と即座に却下する。
設計案を出すときは操作ステップ数を最小化する方向で考える。

> 「うわあ、めちゃくちゃめんどくさい。Web画面で写真を選択したら、それをChatGPTに送って…」

### ルールベースよりAI処理を好む

単純な文字列連結などのロジックには「AIでいい感じに作ることはできないか」と聞いてくる。
OpenAI API や Claude API を使った意味的処理を積極的に選ぶ。

### 正確性・トレーサビリティを最優先にする

「間違った橋脚に間違った所見が保存される」リスクを最大の懸念として挙げた。
利便性より正確性・追跡可能性を優先する場面が多い。

### 課題の制約（カスタムGPT使用）を前提にする

学習課題の要件として「カスタムGPTを使う」が前提にある。
そのため ChatGPT を完全に捨てる設計変更には踏み込まない。

## コミュニケーションパターン

### 「これ全然違う」と明確に言う

方向性がずれているときははっきり言う。
この場合は謝罪より「何が違ったか」の確認を優先する。

### 「ほかにないか？」と深掘りを求める

提案が一通り出た後に「他に何かないか」と聞いてくる。
自分で気づいていない観点を引き出すことに積極的。

### ステップバイステップの説明を求める

エンジニアではないため、PowerShell の操作手順など
「何をどの順番でやるか」を具体的に書く必要がある。

### ChatGPT にも聞いて比較する

重要な設計判断については ChatGPT にも同じ質問を投げて比較することがある。
Claude の回答と ChatGPT の回答が食い違った場合は、より詳しい根拠が必要。

## 技術レベル・前提知識

- エンジニアではないが、システムの概念理解は早い
- Vibe Coding（AIにコードを書かせて動かす）スタイルで開発している
- PowerShell は使えるが、コマンドの意味の説明が必要
- GAS の再デプロイ手順など、GUI 操作のステップ説明が必要

## このプロジェクトでの繰り返しパターン

以下は「言わなくてもわかってほしかった」ことが繰り返されたもの。
次回は先回りして対応する。

| 繰り返しパターン | 先回りすべき対応 |
|---|---|
| 「全文出して」と言われた | 常に完成版ファイルを出す。差分提示しない |
| 選択肢の補足を求めた | 最初から各選択肢にメリデメを付ける |

| ルールベースよりAI処理を希望した | AI 処理が可能な場合は先にその選択肢を出す |
