# Manus用スライド生成プロンプト（短縮版）

以下の条件で、日本語のプレゼンスライドを作成してください。

## 発表条件
- 発表時間は約10分
- 全体は14枚構成
- 章立ては必ず以下を守ること
  1. 企画の概要
  2. 作成したGPTsについて
  3. デモンストレーション
  4. 最終成果物作成の振り返り
  5. 今後に向けて

## 重要な方針
- 主役はGPT単体ではなく、業務システム全体
- ただし一番共有したい学びは、  
  **「理想のUXを目指したが、GPTsの制約にぶつかり、回避策を設計したこと」**
- 強調したい価値は「地図で一覧で見られること」「報告書作成が楽になったこと」
- デモは、写真登録・地図表示・報告書作成まで全部見せる
- スライド1枚あたりの文字数は多すぎないようにする
- 箇条書き中心で、発表時に読み上げやすい表現にする
- 見出しと本文の階層を明確にする
- As-Is / To-Be は1枚に上下配置する
- 「苦労したこと」と「気づき・学び」は1枚に左右配置する

## スライド構成

### 1. タイトル
- 高架下点検マップ＆報告書システムの開発
- 理想のAI体験を目指し、GPTsの制約と回避策を学んだ実装
- キカガク最終発表
- 石丸 浩司

### 2. ① 企画の概要：テーマ・背景
- ChatGPTに高架下点検写真を入れるだけで、整理・保存・報告書作成までつながる体験を目指した
- 写真・所見・位置情報・報告書が分散しやすく、整理や報告書作成が手間だった
- 「写真を入れるだけでDBができていくUX」を実現したかった
- しかし、実際に試す中でGPTsには無視できない制約があると分かった
- 今回一番共有したいのは、システムそのものよりも、この学びである

### 3. ① 企画の概要：目的・ゴール
- AIを用いて、高架下点検写真を起点に、整理・保存・可視化・報告書作成までをなめらかにつなぐことが目的
- 理想は、ChatGPTに写真を入れるだけでDB化し、そのまま地図表示や報告書作成につながること
- 実際のゴールは、理想UXがどこまで実現可能か検証し、制約を特定し、回避策を設計すること
- 最大の評価軸は、GPTsの制約を見極め、実用的な設計に落とし込めたかどうか

### 4. ② 作成したGPTsについて：最初に目指した構想
- 写真をそのままChatGPTに入れ、橋脚番号・所見・位置情報・撮影日時を理解させたかった
- 自動でデータベース化し、地図表示や報告書作成までつなげたかった
- 理想UXは「写真を入れるだけで整理が進む」こと
- しかし、この理想はGPTsの制約によってそのままでは実現できなかった

### 5. ② 作成したGPTsについて：制約を踏まえて再設計した構成
- GPTsはEXIFの位置情報や撮影日時を直接扱えない
- 複数画像では photo_id と画像の対応がずれるリスクがある
- 回避策として、先にWeb画面で写真登録・EXIF取得・photo_id確定を行う形にした
- その後、ChatGPTに画像と対応済みJSONを渡し、GPTは提案と保存支援に役割を絞った
- 加えて、保守性確保のため、管理者向け・運用担当者向けガイドも整備した
- GitHubを、コードだけでなく設計判断・思考プロセス・引き継ぎ資料を残す場として活用した

### 6. ② 作成したGPTsについて：業務フロー図（As-Is / To-Be）
As-Is / To-Be の業務フロー図は、別途添付する参考画像（P8抜粋）のような見せ方にしてください。
- As-Is を上段、To-Be を下段に配置
- 各段はシンプルな横流れフロー
- 1段あたり4〜5個程度の箱と矢印
- 余計な装飾や複雑な図解は不要
- それぞれのフローの下に、課題または改善ポイントを短く添える
- 参考画像はレイアウトの見せ方だけを真似してください。
- 中身の文言は今回の発表内容に合わせて差し替えてください。

#### As-Is
- 写真を取得
- 内容を確認
- 橋脚ごとに整理
- 所見をまとめる
- 報告書に転記
- 全体像は個別ファイルや人の記憶に依存

#### To-Be
- 写真をWeb画面から登録
- EXIFや位置情報を先に取得
- photo_idを確定したうえでGPTに渡す
- GPTが所見・タグ・対応要否を提案
- 既存pin照合や新規pin作成を支援
- 地図上で橋脚単位に一覧化
- pinを選んで報告書作成、PDF出力

### 7. ② 作成したGPTsについて：制約と回避策から得た設計ポイント
- GPTsでは位置情報と撮影日時を直接取り込めない
- 画像とID対応が崩れると業務記録の信頼性が落ちる
- だからAIに全部任せず、AIに任せる範囲を明確にする必要がある
- 回避策は妥協ではなく、実用性を高める設計判断である
- 作るだけでなく、保守・引き継ぎ・判断の履歴を残すことも重要だった
- さらに、最初に仕様書を練り上げ、ChatGPTで作成 → ClaudeとGeminiでレビュー → ChatGPTで再整理する流れにしたことで、後のやり直しを減らせた
- Vibe coding の良さはあるが、最初にきちんと決めることの重要性を学んだ

### 8. ③ デモンストレーション：デモの流れ
- 写真登録
- EXIF取得とphoto_id付きデータ生成
- 地図上で橋脚単位に表示
- 写真詳細の確認・編集
- 報告書作成
- PDF生成まで
- 見せたいのは、GPT単体ではなく、再設計した業務フロー全体

### 9. ③ デモンストレーション：写真登録
- 写真アップロード
- EXIF読み取り
- GAS登録
- photo_id付きで管理対象化
- 本当はここもChatGPTだけでやりたかったが、制約のためWeb画面側で先に処理している

### 10. ③ デモンストレーション：地図・詳細表示
- 地図上でpin表示
- 色や未報告バッジで状況を可視化
- 右パネルで写真詳細確認・編集
- 地図で一覧で見られることが大きな業務価値
- 写真1枚単位の情報と橋脚単位の管理を両立している

### 11. ③ デモンストレーション：報告書作成
- 対象pinを選ぶ
- 実施日・場所・内容を提案
- 報告書作成
- PDF生成
- 回避策を経由しても、最終的には実務に近い体験を実現できた

### 12. ④ 最終成果物作成の振り返り：成果・価値
- 写真・所見・位置情報・報告書を一つの流れで扱えるようになった
- 地図上で点検対象ごとの情報を一覧化できるようになった
- 報告書作成までつながり、実務での利用イメージが持てる形になった
- GPTを単なる会話ではなく、業務支援の実装として組み込めた

### 13. ④ 最終成果物作成の振り返り：苦労したこと / 気づき
#### 苦労したこと
- GPTに任せる範囲と人が確認する範囲の切り分け
- 画像とphoto_idの対応ずれを防ぐ設計
- 理想UXをそのまま実現できないと分かった時の割り切り
- 保守や引き継ぎまで見据えたドキュメント整備

#### 気づき・学び
- GPTは単体で使うより、業務の流れの中に入れた方が価値が大きい
- AIの限界を知ること自体が重要
- 回避策は実運用に耐えるための重要な設計判断
- GitHubはノンエンジニアでも思考プロセスや判断を体系的に蓄積できる
- キカガクで学んだことをすぐ実務で試すことで、自分の仕事の進め方が大きく進化した
- 特に大きかったのは、Claudeを使い始めたことと、GitHubを使い始めたことだった
- 今回一番共有したい学びは、理想のUXを描くだけでなく、AIの制約を見極めて設計を変えることの重要性だった

### 14. ⑤ 今後に向けて
#### このアプリで今後取り組みたいこと
- アップロードからGPT処理までをよりシームレスにする
- pin再整理やフィルタ機能を強化する
- 報告書との差分表示や経年変化の把握にも広げる
- 実運用で工数削減効果を測定する
- 将来的には、位置情報や撮影日時まで自然に扱えるAI基盤が使えれば、より理想のUXに近づけたい

#### 自分自身として今後取り組みたいこと
- GitHubを、非エンジニアの知的資産や意思決定を蓄積する基盤としてさらに活用したい
- 今回のプレゼン資料づくり自体にも GitHub を活用し、成果物と学びを再利用可能な形で残したい
- キカガクで得たヒントを今後もすぐ実務で試し、自分の仕事の進め方そのものを進化させていきたい

## デザイン方針
- シンプルでビジネス向け
- 白背景または淡色背景
- 重要語句は太字
- 1枚の情報量は多すぎない
- As-Is / To-Be は上下2段
- 苦労したこと / 気づき は左右2カラム
- デモ3枚はスクリーンショット中心
- 文章をそのまま大量に載せず、図解・フロー・比較表を優先する
