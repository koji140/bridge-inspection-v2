# Manus再投入用プロンプト（超圧縮版）

以下の条件で、日本語のプレゼンスライドを作成してください。

## 最重要ルール
- 原稿の文章をそのままスライドに貼り付けないでください
- 各スライドは「見出し + 3〜5個の短い bullet」に要約してください
- 1つの bullet は最大2行までにしてください
- 長い段落は禁止です
- 1枚につき伝えたいことは1つに絞ってください
- 詳細説明は削って構いません。発表者が口頭で補足します
- 文章量が多い場合は、箇条書きを減らし、図・比較・フローに置き換えてください
- 余白を十分に確保してください
- 文章をそのまま大量に載せず、図解・フロー・比較表を優先してください

## テンプレート
- 添付した PowerPoint テンプレート「【テンプレート】生成AIコース最終発表（お名前）.pptx」を必ずベースに使用してください
- 新しい独自デザインは作らないでください
- 配色・フォント・レイアウト方針はテンプレート準拠にしてください
- 名前は「石丸浩司」にしてください

## 発表条件
- 発表時間は約10分
- 全体は14枚構成
- 章立ては必ず以下を守ること
  1. 企画の概要
  2. 作成したGPTsについて
  3. デモンストレーション
  4. 最終成果物作成の振り返り
  5. 今後に向けて

## 発表全体の主メッセージ
- 主役はGPT単体ではなく、業務システム全体
- ただし一番共有したい学びは、
  「理想のUXを目指したが、GPTsの制約にぶつかり、回避策を設計したこと」
- 強調したい価値は「地図で一覧で見られること」「報告書作成が楽になったこと」
- 用語は「高架下点検」に統一
- workaround という英語は使わず、「回避策」と表現する

## スライドの型
- 1〜5枚目: タイトル + 要点3〜4個
- 6枚目: As-Is / To-Be の上下比較フロー図
- 7枚目: 左右2カラム（制約 / 設計ポイント）
- 8枚目: デモ全体フロー
- 9〜11枚目: スクリーンショット大 + 要点3個
- 12枚目: 成果を4点
- 13枚目: 左右2カラム（苦労 / 学び）
- 14枚目: 2セクション（このアプリ / 自分自身）

## As-Is / To-Be フロー図
- 別途添付する P8 抜粋画像の見せ方を参考にしてください
- As-Is を上段、To-Be を下段に配置
- 各段はシンプルな横流れフロー
- 1段あたり4〜5個程度の箱と矢印
- 余計な装飾や複雑な図解は不要
- 参考画像はレイアウトだけ参考にし、中身の文言は今回の発表内容に合わせて差し替える

## デモスライドの扱い
- 9〜11枚目のデモ3枚（写真登録 / 地図・詳細表示 / 報告書作成）は、後から私が実際のスクリーンショットを貼ります
- そのため、この3枚は
  - スライドタイトル
  - 見せるポイント
  - 短い補足文
  - 大きなスクリーンショット枠
 だけを作成してください
- デモ画面の内容を新たに考案したり、仮画像・生成画像を入れたりしないでください

## 各スライドの中身

### 1. タイトル
- 高架下点検マップ＆報告書システムの開発
- 理想のAI体験を目指し、GPTsの制約と回避策を学んだ実装
- キカガク最終発表
- 石丸浩司

### 2. ① 企画の概要：テーマ・背景
- ChatGPTに高架下点検写真を入れるだけで、整理・保存・報告書作成までつながる体験を目指した
- 写真・所見・位置情報・報告書が分散しやすく、整理や報告書作成が手間だった
- 特に、地図で一覧しながら状況を把握したい思いがあった
- そのため、位置情報を活用できることが重要だった
- 今回一番共有したいのは、システムそのものよりも、この学びである

### 3. ① 企画の概要：目的・ゴール
- 目的は、高架下点検写真を起点に、整理・保存・可視化・報告書作成までをなめらかにつなぐこと
- 理想は、ChatGPTに写真を入れるだけでDB化し、そのまま地図表示や報告書作成につながること
- 実際のゴールは、理想UXの実現可能性を検証し、制約を特定し、回避策を設計すること
- 評価軸は、GPTsの制約を見極め、実用的な設計に落とし込めたかどうか

### 4. ② 作成したGPTsについて：最初に目指した構想
- 写真をそのままChatGPTに入れ、所見・位置情報・撮影日時まで理解させたかった
- 自動でデータベース化し、地図表示や報告書作成までつなげたかった
- 理想UXは「写真を入れるだけで整理が進む」こと
- しかし、この理想はGPTsの制約によりそのままでは実現できなかった

### 5. ② 作成したGPTsについて：制約を踏まえて再設計した構成
- GPTsはEXIFの位置情報や撮影日時を直接扱えない
- 複数画像では photo_id と画像の対応がずれるリスクがある
- 回避策として、先にWeb画面で写真登録・EXIF取得・photo_id確定を行う形にした
- その後、ChatGPTに画像と対応済みJSONを渡し、GPTは提案と保存支援に役割を絞った
- 管理者向け・運用担当者向けガイドも整備し、保守性も意識した

### 6. ② 作成したGPTsについて：業務フロー図（As-Is / To-Be）
#### As-Is
- 写真取得
- 手動整理・突合
- 所見作成・報告書編集
- PDF化・共有

#### To-Be
- Web登録
- GPT提案
- 地図確認
- 報告書作成・PDF出力

### 7. ② 作成したGPTsについて：制約と回避策から得た設計ポイント
#### 左カラム：制約
- GPTsでは位置情報と撮影日時を直接取り込めない
- 画像とID対応が崩れると業務記録の信頼性が落ちる
- AIだけで完結させると危ない

#### 右カラム：設計ポイント
- AIに全部任せず、AIに任せる範囲を明確にする
- 確定情報は先に別経路で取得する
- 回避策は妥協ではなく、実用性を高める設計判断
- 最初に仕様書を練り上げることで、後のやり直しを減らせた

### 8. ③ デモンストレーション：デモの流れ
- 写真登録
- EXIF取得と photo_id 付きデータ生成
- 地図上で表示・確認
- 報告書作成
- PDF生成
- 見せたいのは、回避策を前提に再設計した業務フロー全体

### 9. ③ デモンストレーション：写真登録
- 写真アップロード
- EXIF読み取り
- GAS登録
- photo_id付きで管理対象化
- ここは大きなスクリーンショット枠を配置

### 10. ③ デモンストレーション：地図・詳細表示
- 地図上でpin表示
- 色や未報告バッジで状況を可視化
- 右パネルで写真詳細確認・編集
- 地図で一覧で見られることが大きな業務価値
- ここは大きなスクリーンショット枠を配置

### 11. ③ デモンストレーション：報告書作成
- 対象pinを選ぶ
- 実施日・場所・内容を提案
- 報告書作成
- PDF生成
- ここは大きなスクリーンショット枠を配置

### 12. ④ 最終成果物作成の振り返り：成果・価値
- 写真・所見・位置情報・報告書を一つの流れで扱えるようになった
- 地図上で点検対象ごとの情報を一覧化できるようになった
- 報告書作成までつながり、実務での利用イメージが持てる形になった
- GPTを単なる会話ではなく、業務支援の実装として組み込めた

### 13. ④ 最終成果物作成の振り返り：苦労したこと / 気づき
#### 左カラム：苦労したこと
- GPTに任せる範囲と人が確認する範囲の切り分け
- 画像とphoto_idの対応ずれを防ぐ設計
- 理想UXをそのまま実現できないと分かった時の割り切り
- 保守や引き継ぎまで見据えたドキュメント整備

#### 右カラム：気づき・学び
- GPTは単体で使うより、業務の流れの中に入れた方が価値が大きい
- AIの限界を知ること自体が重要
- GitHubはノンエンジニアでも思考プロセスや判断を体系的に蓄積できる
- キカガクで学んだことをすぐ実務で試すことで、自分の仕事の進め方が大きく進化した
- 特に大きかったのは、Claudeを使い始めたことと、GitHubを使い始めたことだった

### 14. ⑤ 今後に向けて
#### このアプリで今後取り組みたいこと
- アップロードからGPT処理までをよりシームレスにする
- pin再整理やフィルタ機能を強化する
- 報告書との差分表示や経年変化の把握にも広げる
- 実運用で工数削減効果を測定する

#### 自分自身として今後取り組みたいこと
- GitHubを、非エンジニアの知的資産や意思決定を蓄積する基盤としてさらに活用したい
- プレゼン資料づくり自体にも GitHub を活用し、成果物と学びを再利用可能な形で残したい
- キカガクで得たヒントを今後もすぐ実務で試し、自分の仕事の進め方そのものを進化させていきたい
- 最後に、「理想と制約の両方を理解し、現場で使える形に再設計することがAI活用では重要だった」という締めメッセージを入れてください
